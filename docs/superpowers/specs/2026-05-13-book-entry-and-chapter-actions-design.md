# Book Entry and Chapter Actions Design

## Context

当前 `fanqie-workbench` 已经把 `BooksPage` 提升成默认首页，但“怎么开始写一本新书”仍然不清晰，用户看到的入口仍然偏工具化：

- 自由会话仍然可见，容易被误解为主入口
- `oh-story-claudecode` 与 `chinese-novelist` 的关系对用户不透明
- 高级功能没有明确分层，用户不知道这些能力应该作用在整本书还是某一章
- 执行结果虽然已有 session / stream 基础设施，但产品心智还没有收束成“在某本书里持续推进写作”

用户已经明确给出以下产品判断：

1. **默认主流程应该走 `oh-story-claudecode`**，减少 token 浪费和入口分流。
2. **`chinese-novelist` 不应抢主入口**，但要保留在前端可见范围内，供用户在写到一半时调用其中某些能力。
3. **高级功能更多是章节级的**，不是书级工具堆叠。
4. **流式输出必须保留**，并且统一展示在右侧面板。

因此，这次设计的目标不是再增加一个“执行入口”，而是把“开一本书”和“处理某一章”整理成清晰的产品路径。

## Goals

### Primary goals

1. 用户第一次进入产品时，能直接理解“从哪里开始写一本新书”。
2. 用户推进一本书时，默认只需要使用主流程，不需要理解 skill 名字。
3. 用户需要精修时，可以在某一章上明确找到高级能力。
4. 所有动作都复用统一的 session / streaming 面板，不再分散到不同页面心智里。

### Non-goals

1. 本设计不改变底层 session 流式机制。
2. 本设计不重新定义 `oh-story-claudecode` 或 `chinese-novelist` 的内部 prompt。
3. 本设计不引入多套并行主流程。
4. 本设计不在这一轮解决所有书级工具的具体实现细节。

## User Experience Model

## 1. Main entry: create a new book

`BooksPage` 应当成为唯一主入口。顶部最重要的 CTA 不是“自由会话”，而是：

- **新建一本书**

点击后进入一个轻量开书流程。该流程默认触发 **`oh-story-claudecode`**，用户不需要手动选择 skill。

用户输入的不是复杂 prompt，而是一句自然语言想法，例如：

- “我想写一本女频悬疑复仇文，现代都市，强反转”
- “男频诡异修仙，前期压抑，中后期爆发”

系统在后台基于这句输入生成：

- 书名
- 简介
- 大纲
- 初始章节目录

完成后自动创建书籍目录与数据库记录，并把用户带入该书的工作台。

### Why this is the right default

- 用户真正的目标是“开始一本书”，不是“开一个 prompt 会话”
- `oh-story-claudecode` 更适合作为整本书主流程引擎
- 默认不暴露 skill 选择，能显著降低理解成本

## 2. Page hierarchy inside a book

进入某一本书后，页面分成三块心智区域：

### A. Book-level summary

显示当前书的整体状态，例如：

- 总章节
- 待写作数量
- 可发布数量
- 已发布数量
- 当前是否存在活跃会话

它只回答一个问题：

> 这本书现在整体进展到哪了？

### B. Chapter list as the real action surface

章节列表是用户真正高频操作的区域。每章只暴露两个层级：

- **处理**：默认主流程
- **`···`**：章节级高级功能

这样用户看到章节时不需要理解一堆按钮，只要判断：

- 我要继续正常推进这一章 → 点 `处理`
- 我要对这一章做特殊操作 → 点 `···`

### C. Right-side streaming panel

页面右侧固定一个流式会话面板，统一展示 Claude 正在做什么。

无论用户触发的是：

- `处理`
- `··· > 润色`
- `··· > 去AI味`
- `··· > 审稿`
- `··· > 重写`

都统一在这个右侧面板里输出。

用户不应该被带回“自由会话”页，也不应该跳转到另一套执行视图。

## 3. Chapter action model

### Default action: 处理

`处理` 是主流程按钮，固定走 **`oh-story-claudecode`**。

它用于：

- 推进待写作章节
- 按既定 pipeline 继续该章状态流转
- 正常情况下成为用户最常使用的按钮

这个按钮代表的是：

> 按主流程继续写这一章。

### Advanced action: `···`

`···` 是章节级高级入口。这里才出现更细分的能力，例如：

- 润色
- 去AI味
- 审稿
- 重写本章
- 回退本章
- 调用 `chinese-novelist` 的某个章节能力补做

这套菜单的核心原则是：

- **高级能力不抢主流程按钮的位置**
- **高级能力默认按“章节”挂载，而不是按“全局工具箱”挂载**

## 4. Role of chinese-novelist

`chinese-novelist` 不应成为开书主流程的一部分。

推荐定位：

- 前端保留可见性，但不放在主入口
- 只出现在章节 `···` 菜单或高级能力区域
- 作为“写到一半补用某个能力”的补充 skill

这能满足两个目标：

1. 保留它的能力价值
2. 不让它与 `oh-story-claudecode` 争抢默认心智

## 5. Role of free chat

“自由会话”应明确降级为辅助页，用途包括：

- 临时试验
- 开书前脑暴
- 调试 skill 行为
- 排查 session 问题

它不承担主工作流。文案与导航都要明确这一点。

## 6. Role of book-level tools

虽然高级能力主要是章节级的，但仍然有一小部分功能应保留为**书级功能**。这些功能必须真的是“整本书维度”，例如：

- 生成封面
- 调整大纲
- 导入小说
- 全书分析

其中，封面应被视为**书级功能**，而不是章节高级功能。

但这些书级工具不应该占据主视觉，不应和“处理章节”同级竞争。它们更适合放在书摘要附近的次级工具区或书级下拉菜单内。

## Interaction Flows

## Flow 1: Start a new book

1. 用户进入 `BooksPage`
2. 点击 `新建一本书`
3. 输入一句开书需求
4. 前端创建一个 book-creation session，默认指定 `oh-story-claudecode`
5. 右侧流式面板开始输出：构思 / 书名 / 简介 / 大纲 / 章节目录
6. 完成后自动创建书并选中它
7. 页面切入该书工作台

## Flow 2: Continue a chapter normally

1. 用户在书里找到某章
2. 点击 `处理`
3. 后台创建 chapter session，默认主流程继续推进
4. 右侧流式面板展示该章执行过程
5. 若有 question，直接在右侧面板完成 human-in-the-loop
6. 完成后章节状态与书摘要同步更新

## Flow 3: Use advanced chapter action

1. 用户在某一章点击 `···`
2. 选择一个动作，例如“润色”
3. 前端用该动作创建对应 session / action context
4. 右侧面板统一流式展示执行过程
5. 完成后章节更新，但用户仍停留在当前书工作台

## UI Structure

## BooksPage top level

推荐结构：

- 顶部：页面标题 + `新建一本书`
- 左 / 中主区域：
  - 书列表
  - 当前书摘要
  - 章节列表
- 右侧固定面板：
  - 当前动作标题
  - 当前章节
  - 流式日志
  - 提问 / 回答
  - 状态 badge

## Chapter row

每章显示：

- 章节标题
- 当前状态
- `处理`
- `···`

不要在章节行直接摊开一排高级按钮。

## Data and Routing Implications

本设计不要求彻底重做后端模型，但前端与 session 路由需要形成更清晰约定：

### Session kinds / intents

对于章节操作，前端应明确区分“主流程”与“高级动作”的 intent，例如：

- `chapter-process`
- `chapter-polish`
- `chapter-deslop`
- `chapter-review`
- `chapter-rewrite`

不一定都需要新的 `kind` 字段，但至少要能在前端和日志面板中清楚表达“当前动作是什么”。

### Book creation session

需要引入一个“开书 session”概念，至少让前端能用统一流式面板展示：

- 正在生成书名
- 正在生成简介
- 正在生成大纲
- 正在创建章节目录

这可以是新的 session kind，也可以是 `prompt` 的受约束形态；但前端心智必须是“新建一本书”，不是“执行一条 prompt”。

### Book-entry progress presentation

开书过程不应只显示一段原始日志，而应有明确的产品级展示结构。

推荐结构：

1. **固定 6 步进度**
   - 生成书名
   - 生成简介
   - 生成大纲
   - 生成章节目录
   - 创建书籍
   - 进入工作台

2. **完整结果展示区**
   每个阶段拿到什么，就直接展示完整内容，而不是先做摘要提炼。例如：
   - 书名全文
   - 简介全文
   - 大纲全文
   - 章节目录全文

3. **原始流式日志区**
   保留调试与透明度价值，但视觉优先级应低于阶段进度和阶段结果。

### Why fixed product stages

这 6 步是“开一本书”的产品阶段，不是某个具体 skill 的内部阶段。

这样即使未来替换 `oh-story-claudecode` 为别的更强开书引擎，前端仍然可以保持稳定的进度展示，不需要重做整套 UI。

### Why full content instead of summaries

当前阶段直接展示完整内容更合适：

- 实现简单，减少前端额外提炼逻辑
- 不会因为摘要规则不准而误导用户
- 用户在开书时更关心“它具体写出了什么”，不是一句抽象标签

## Error Handling

- 若开书失败，仍停留在开书弹层 / 面板中，显示失败原因与重试入口
- 若章节高级动作失败，不跳页，右侧面板直接显示失败状态
- 若 question 等待回答，状态必须在章节列表和右侧面板都有明确提示

## Testing Strategy

### UI tests

至少覆盖：

1. 默认首页为 `BooksPage`
2. `新建一本书` 入口存在且优先级最高
3. 某章默认存在 `处理` 与 `···`
4. `···` 展开后出现章节级高级功能
5. 触发动作后右侧面板能显示对应动作与流式状态
6. `自由会话` 文案明确为辅助页

### Route / integration tests

至少覆盖：

1. 开书动作创建 session 并能流式输出
2. 章节高级动作能被区分为不同 intent
3. 右侧面板恢复时能识别当前书 / 当前章 / 当前动作

## Recommended Implementation Order

1. 在 `BooksPage` 顶部增加 `新建一本书` 入口
2. 抽象“开书 session”前端状态与流式展示
3. 为章节行增加 `···` 高级入口
4. 把章节级高级动作路由到统一 session 面板
5. 把 `chinese-novelist` 收到章节高级功能里
6. 把封面等真正书级功能放到书级工具区
7. 继续弱化 `自由会话` 的主入口感

## Final Product Rule

整个产品要让用户形成的心智是：

- 我要开始一本新书 → 去书籍管理，点新建一本书
- 我要继续写某一章 → 点处理
- 我要对某一章做特殊精修 → 点 `···`
- 我要看 Claude 正在干什么 → 看右侧流式面板

用户不需要理解 skill 名字、prompt 组织方式或底层 session 模型，系统负责在后台默认触发正确能力。
