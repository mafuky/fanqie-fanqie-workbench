# Book-Level Session Design

## Context

当前 `fanqie-workbench` 已经初步具备：

- `BooksPage` 作为主入口
- “新建一本书”入口
- 章节 `处理` 与 `···` 高级动作
- 页面级执行器与流式日志

但用户已经明确指出一个更底层的问题：

> 不应该每次动作都起一个新的 Claude。

如果一本书的开书、规划、方向修正总是依赖一次性新会话，会出现：

- 上下文无法持续累积
- token 反复浪费
- 书整体风格和长期约束不稳定
- 用户无法理解“系统到底记住了什么”

因此，需要给“整本书”建立一条长期主会话，而不是继续把所有事情都建模为独立一次性任务。

## Goals

1. 为每本书建立一条长期主会话，用于承载整本书级别的持续上下文。
2. 将“书级主会话”与“章节处理会话”分层，而不是把所有事塞进同一条执行流。
3. 让前端可见地管理这条主会话，包括查看状态、查看上下文、手动压缩上下文。
4. 让后端持有并复用 Claude resume 标识，但不把技术细节强行暴露为用户主界面元素。

## Non-goals

1. 本设计不要求所有章节级动作立即合并进长期主会话。
2. 本设计不要求前端直接显示原始 Claude slash commands。
3. 本设计不要求本轮就实现完整的上下文可视化 diff 工具。

## Core Model

## 1. One long-lived session per book

每本书应有且仅有一条**书级长期主会话**。

这条会话主要服务于：

- 开书
- 大纲
- 书整体方向调整
- 设定延续
- 长期上下文沉淀

它不是“某一章的写作流水线”，而是：

> 这本书的主脑 / 长期规划上下文

## 2. Chapter processing remains separate for now

用户已经确认：

- **书级主会话**负责开书与整体规划
- **章节处理**暂时保持独立会话

因此当前推荐模型是：

- 书级主会话：长期保留、可恢复、可压缩
- 章节处理会话：按章启动、按章完成

这样比“所有章节都强绑进一个主会话”更稳，也更容易逐步迁移。

## 3. Resume identifier stays backend-owned

后端需要保存 Claude 的 resume 会话标识，以便：

- 继续这本书的主会话
- 页面刷新后恢复这本书的上下文链路
- 后续执行“压缩上下文”后仍能继续同一条主线

但这个 resume id 的默认定位是：

- **后端内部恢复字段**
- 默认不作为主界面字段高调展示

用户层看到的是：

- 这本书是否已有主会话
- 当前是否正在使用主会话
- 主会话最近更新时间
- 主会话是否做过压缩

## User Experience Model

## 1. Book-level session panel

每本书的工作台里增加一个**书级主会话面板**。

默认展示内容采用“标准版”：

- 当前主会话状态
- 当前使用中的书级 skill / 阶段
- 最近一次更新时间
- 是否已压缩过
- `压缩上下文` 按钮
- `查看上下文` 入口

这个面板的目的不是调试 CLI，而是回答：

- 这本书现在有没有主脑？
- 它最近在干什么？
- 我能不能手动压缩？
- 我能不能看看它记住了什么？

## 2. View context

用户已确认：希望能查看该会话的 context。

推荐提供两层查看方式：

### A. 用户可读版 context

这是默认入口，面向创作使用。展示内容例如：

- 当前书名 / 题材
- 世界观与主要设定
- 核心人物状态
- 当前大纲摘要
- 最近几章进展
- 当前 Claude 记住的工作重点

### B. 原始上下文全文

作为展开后的技术视图，允许用户查看更底层的 prompt / context 结构。

这样既满足“可见”，也不会默认把技术细节塞满主界面。

## 3. Manual compaction only

用户明确偏好：

- **不要偷偷自动压缩**
- 必须**手动档**
- 而且这个动作要在前端显式可见

因此本设计规定：

- 书级主会话默认不自动压缩
- `压缩上下文` 是书级主会话面板内的显式按钮
- 压缩完成后，需要更新“最近压缩时间 / 是否已压缩”状态

这里的核心不是实现某个具体 CLI 命令，而是产品动作：

> 用户主动要求为这本书压缩上下文。

## 4. Backend command mapping

前端不直接暴露 Claude 原始命令。用户只看到产品动作，例如：

- 新建一本书
- 调整大纲
- 压缩上下文
- 查看上下文

后端负责把这些动作翻译成：

- 具体 skill 调度
- resume 会话恢复
- 会话压缩命令

这保证前端体验是产品化的，而不是把 CLI 指令硬搬上来。

## Recommended Session Ownership

### Book-level actions

这些动作优先绑定到书级主会话：

- 新建一本书
- 调整大纲
- 书整体方向修正
- 设定修订
- 长期规划
- 压缩上下文

### Chapter-level actions

这些动作当前保持为独立章节会话：

- `处理`
- 润色
- 去AI味
- 审稿
- 重写本章

后续如果要继续增强，也可以考虑让章节会话在启动时读取书级主会话的摘要上下文，而不是完全孤立。

## Data Implications

为支持书级长期主会话，需要至少补以下能力：

1. `sessions` 需要能表达“这是书级主会话”
2. 需要持久化 Claude resume 标识
3. 需要记录最近压缩信息
4. 需要能区分：
   - 书级主会话
   - 章节处理会话
   - 章节高级动作会话

推荐不要把这些全挤进 `currentSkill` 字段里糊成一团，而是明确区分会话用途。

## Suggested Frontend Structure

在 `BooksPage` 的书工作台中：

- 书摘要区
- 书级主会话面板
- 章节列表
- 右侧流式执行器

其中：

- 右侧执行器继续承担“当前正在跑的事”
- 书级主会话面板承担“这本书的长期记忆和恢复能力”

这两者不能混为一谈。

## Error Handling

- 若书级主会话 resume 失败，应明确提示“主会话恢复失败”，而不是静默退回为新会话
- 若压缩失败，应保留原会话状态，不要误标成“已压缩”
- 若查看上下文失败，应允许用户重试，不影响章节处理主路径

## Testing Strategy

至少覆盖：

1. 一本书创建后可持久化主会话身份
2. 继续书级动作时会复用主会话，而不是新建孤立 prompt 会话
3. 前端能显示书级主会话面板
4. 前端有可见的 `压缩上下文` 按钮
5. 手动压缩后状态可见更新
6. `查看上下文` 至少能打开用户可读版视图
7. 章节处理不因为引入书级主会话而回归破坏

## Final Rule

整本书的长期规划与上下文，不应继续建模为“一次性新 Claude”。

正确心智是：

- 一本书有一条长期主会话
- 章节处理暂时独立
- 前端能看见并管理主会话
- 上下文压缩必须是手动、可见、可控的
